# QUALITY.md

> 定义项目质量门槛。让人和 AI 都知道什么叫"可以合并"。

## 合并前最低要求

```bash
# 必须通过
pnpm lint          # 代码规范检查
pnpm typecheck     # 类型检查
pnpm test          # 单元测试

# 特定场景必须通过
pnpm test:e2e      # 核心流程改动时
```

## 通用质量标准

### 性能
- 首屏 LCP < 2.5s
- 交互响应 < 300ms
- Bundle Size 增量 < 50KB（需说明原因）

### 可访问性
- 表单控件必须有 label
- 键盘可操作（Tab、Enter、Esc）
- 颜色对比度 >= 4.5:1
- 错误提示可理解（不只说"失败"）

### 代码质量
- 无 TypeScript `any`（除非有充分理由）
- 无 ESLint error（warning 需评估）
- 公共函数有注释说明
- 复杂逻辑有单元测试

### 用户体验
- 所有异步操作有 loading 状态
- 所有请求有 error 处理
- 空状态有提示和引导
- 危险操作有二次确认

## 截图验收要求

### 必须截图的场景
- UI 改动后的最终效果
- 核心交互流程的关键步骤
- 错误态、空态、加载态

### 截图验收标准
**功能验收**：交互流程完整可执行、无明显报错

**视觉验收**（必须单独审查）：
- ✅ 样式已生效（非裸 HTML）
- ✅ 布局合理（有层次、间距、对齐）
- ✅ 交互状态完整（hover、focus、disabled、loading）
- ✅ 颜色、字体、间距符合设计规范
- ❌ 若页面接近裸 HTML、无明显层级或布局失衡，不得判定通过

### 截图工具
Playwright 自动化截图 → agent-browser skill → 浏览器开发者工具截图（依次尝试）

## 服务探测流程

```bash
# 1. 检查端口占用
lsof -i :3000 或 netstat -ano | findstr 3000

# 2. 如果端口已占用，检查服务健康
curl http://localhost:3000

# 3. 判断服务状态
# - 端口未占用 → 直接启动
# - 端口占用 + 服务健康 → 复用现有服务
# - 端口占用 + 服务不健康 → 清理后重启（kill <PID> + rm -rf .next + pnpm dev）
```

禁止：不检查就重复启动 / 端口冲突时换新端口 / 认为"端口可访问"等于"服务健康"

## 失败后的重跑与定位

### 测试失败
1. 读取完整错误日志（不只看摘要）
2. 定位原因：代码逻辑 / 环境问题 / 测试用例问题
3. 修复后重新运行所有测试（不只跑失败的）
4. 反复出现的问题 → 更新文档添加约束

### 构建失败
1. 清理构建产物：`rm -rf .next node_modules/.cache`
2. 检查依赖：`pnpm install`
3. 重新构建：`pnpm build`

### E2E 失败
1. 按服务探测流程检查服务状态
2. 查看 Playwright 生成的失败截图和控制台日志
3. 清理环境后重试：清理 `.next` + 重启服务 + 重跑测试

## 测试策略

### 单元测试（Vitest）
- 纯函数、工具函数
- 复杂 hooks
- 业务逻辑计算

### E2E 测试（Playwright）
- 核心用户路径（登录、主要业务流程）
- 跨页面交互
